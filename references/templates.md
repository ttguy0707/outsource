# 委派提示词骨架与检查清单

主线产出委派时复制改造：**一份自包含的五段式提示词**，报告格式**内嵌在"产出"段**，用户一次粘贴即可。按场景的现成模板见 `scenarios/`（目录 `scenarios/index.md`）。

---

## 五段式委派提示词骨架

> 目标 session 常是**低配模型**：写得它**理解意图、不靠猜**，并堵住廉价模型通病（编造、用概括代替原文、跳步、过度自信）。报告格式直接嵌在"产出"段。

```
# 角色
你是<领域>专家。本次任务：<一句话目标>。

# 背景（帮你理解在找什么，不是走流程）
- 对象：<项目绝对路径 / 技术栈 / 版本>
- 为什么做：<意图，让你知道这件事要确认/拿到什么>
- 你会用到：<工具/命令>

# 步骤（按序，每步把真实输出留作证据）
1. <可直接执行的命令>　目的：<这步在确认什么>
2. <命令；若 X 不可用，改用 Y>　目的：<…>
3. <…>

# 铁律
- 只读 + 不改任何文件 + 不提交（除非任务明确允许写）。
- 关键信息**逐字/原文**回（版本号、报错、命中行、符号名…），禁止改写、补全、简写、美化。
- **查不到就说查不到**，绝不编造一个看似合理的答案。
- 不确定 / 有歧义 → **标出来并说明**，不要自己猜着填。
- **只报事实与证据，不下主观结论**。
- 卡点（网络/认证/缺工具/超时）如实报，别反复重试，别擅自扩大范围。

# 产出（你按下面格式回传）
## <任务名> 报告
1. 结论：<PASS/FAIL / 是否 / 数值>
2. <字段二>：______
3. <字段三 · 证据原文>：
   ```
   <粘贴报错原文 / 命中行 / 版本号原文>
   ```
4. 卡点：______（无则写"无"）
```

## 检查清单（落盘/发出前过一遍）

- [ ] 角色定位成「**<领域>专家**」+ 直接交代「本次任务：…」（不写"你是助手/替谁干活"）
- [ ] **背景与意图**写清（低配模型靠这个理解在找什么，而非机械跑）
- [ ] 绝对路径、栈、版本都给了（它无记忆）
- [ ] 每步可直接跑 + 附「目的」，关键步给替代项
- [ ] 铁律齐：只读 / 逐字原文 / 查不到说查不到 / 不确定标注 / 只报事实不下结论 / 卡点如实报
- [ ] **报告格式内嵌在"产出"段**（结论先行 + 要证据原文 + 卡点栏），用户一次粘贴

---

## 迭代委派

一轮报告不够时，**基于上一轮证据再产一份更聚焦的委派提示词**（而非主线自己下场跑）。例：第一轮"列版本"→发现某版本 502→第二轮"查该版本是否真存在 + 重试一次确认是否网关临时故障"。
