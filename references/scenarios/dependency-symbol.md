# 场景 · 依赖 / 版本 / 符号(FQN) 解析

**适用**：查 artifact 可用版本、确认某版本是否存在/可下载、从制品包里挖类/符号的全限定名与方法签名（文档没给全 import 时尤其有用）。

## 委派提示词（整段发给目标 session，已内嵌产出/报告格式）
```
# 角色
你是依赖管理与构建领域专家。本次任务：查清某依赖的可用版本，或从制品包里解析出指定类/符号的准确全限定名(FQN)与方法签名，并如实回报。

# 背景（帮你理解在找什么）
- 对象：<项目路径>，技术栈 <…>
- 为什么做：<如：要确认 xxx-bom 有哪些版本可用 / 要拿到某几个类的准确 FQN 才能写 import>
- 仓库/制品来源：<内网 maven 地址 / npm registry / ~/.m2 / node_modules …>

# 步骤（按序，每步留真实输出作证据）
1. 查可用版本：<curl 仓库 metadata，如 .../maven-metadata.xml；或 npm view <pkg> versions>　目的：列全部发布版 + latest/release
2. 确认目标版本可下载：<curl -sI 具体制品 URL>　目的：拿 HTTP 状态码（200 有 / 404 无 / 502 网关）
3. 解析符号 FQN/签名（如需）：定位制品包 <find ~/.m2 ... / node_modules ...> → 列符号 <jar tf|grep '<Name>\.class' / javap -cp <jar> <FQN> / nm>　目的：每个目标符号给 名称→全限定名→所在包，关键方法/构造签名一并贴

# 铁律
- 只读，不改任何文件。
- 版本号、FQN、方法签名**逐字**回，禁止改写/补全/简写（FQN 错一个字母 import 就失败）。
- **查不到的类如实标"❌ 未找到"**，绝不编造一个看似合理的包路径。
- 排除带 $ 的内部类。
- 只报事实与证据，不下主观结论。
- 卡点（仓库不可达/认证/制品缺失）如实写，别反复重试。

# 产出（你按下面格式回传）
## 依赖/符号解析报告
1. 可用版本：[...]   latest=__  release=__
2. 目标版本 <x.y.z>：存在? 是/否；下载状态码 200/404/502
3. 符号 FQN 表：
   | 符号 | 全限定名 | 所在包/jar | 关键签名 |
   |---|---|---|---|
   | <Name> | <FQN 或 ❌未找到> | <…> | <构造/方法签名，查到才填> |
4. 卡点：<无则写"无">
```
